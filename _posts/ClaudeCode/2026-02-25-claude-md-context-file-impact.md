# Introduction

---

CLAUDE.md 또는 AGENTS.md 같은 프로젝트별 컨텍스트 파일은 AI 코딩 에이전트에게 프로젝트 배경을 알려주기 위한 표준적인 방법으로 자리잡았습니다. 잘 만든 CLAUDE.md는 AI가 프로젝트 컨벤션을 지키게 하고, 반복적인 설명을 줄여줄 것이라는 기대를 받아왔습니다.

그런데 2026년 2월에 발표된 ETH Zurich의 논문 **"Evaluating AGENTS.md: Are Repository-Level Context Files Helpful for Coding Agents?"** (Gloaguen et al., arXiv:2602.11988)에서 흥미로운 결과가 나왔습니다. **LLM이 자동 생성한 컨텍스트 파일은 오히려 AI의 작업 성공률을 떨어뜨리고 비용을 20% 이상 올린다**는 것입니다.

이 글은 해당 논문의 핵심 내용과 실험 결과를 정리하고, 같은 시기에 발표된 반대 결과의 논문과 비교한 뒤, 실무에서 컨텍스트 파일을 어떻게 관리해야 하는지 시사점을 도출합니다.

> **이 글의 대상 독자**: Claude Code, Codex, Cursor 등 AI 코딩 에이전트를 실무에서 사용하면서 CLAUDE.md나 AGENTS.md를 작성해본 경험이 있는 개발자

# 1. 배경: AI 컨텍스트 파일이란?

---

논문을 이해하려면 먼저 AI 에이전트가 정보를 어떻게 받아들이는지 알아야 합니다.

## 컨텍스트의 계층 구조

AI 코딩 에이전트가 처리하는 컨텍스트는 크게 두 계층으로 나뉩니다.

```text
시스템 프롬프트 (System Prompt)
  └── AI 자체 설정 (에이전트의 기본 동작 방식, 사용자가 직접 수정 불가)

사용자 접근 가능한 컨텍스트 (User-accessible Context)
  ├── 메모리 파일 (CLAUDE.md, AGENTS.md 등)  ← 사용자 프롬프트보다 높은 우선순위
  ├── 사용자 프롬프트 (현재 요청)
  └── AI의 이전 답변 (대화 기록)
```

중요한 점은 **메모리 파일(CLAUDE.md 등)이 사용자 프롬프트보다 높은 우선순위**를 갖는다는 것입니다. Claude Code의 경우, 세션이 시작되면 프로젝트 루트의 `CLAUDE.md`, `~/.claude/CLAUDE.md`(글로벌), 그리고 `.claude/` 디렉터리 하위의 설정 파일들을 자동으로 읽어 시스템 프롬프트에 주입합니다. AI는 현재 요청을 처리하기 전에 이 내용을 먼저 반영합니다.

## 컨텍스트 파일의 현재 위상

현재 주요 AI 코딩 에이전트들은 모두 프로젝트 루트에 컨텍스트 파일을 두도록 권장하고 있습니다.

| 에이전트 | 컨텍스트 파일명 | 용도 |
|----------|----------------|------|
| Claude Code | `CLAUDE.md` | 프로젝트 규칙, 빌드 명령어, 코딩 컨벤션 |
| OpenAI Codex | `AGENTS.md` | 리포지토리 수준 에이전트 지시사항 |
| Cursor | `.cursorrules` | 프로젝트별 AI 동작 규칙 |
| GitHub Copilot | `.github/copilot-instructions.md` | 코드 생성 지시사항 |

이 구조가 왜 문제가 될 수 있는지, 논문의 실험 결과를 살펴봅니다.

# 2. 논문의 핵심 발견: LLM 자동 생성 컨텍스트 파일의 함정

---

## 실험 설계

Gloaguen et al.의 논문은 두 가지 벤치마크에서 실험을 진행했습니다.

| 벤치마크 | 구성 | 컨텍스트 파일 유형 |
|----------|------|-------------------|
| **SWE-bench Lite** | 11개 인기 리포지토리, 300개 태스크 | LLM이 자동 생성한 파일 |
| **AGENTbench** (논문에서 새로 구축) | 12개 Python 리포지토리, 138개 인스턴스 | 개발자가 직접 작성한 파일 |

테스트에 사용된 AI 모델은 **Claude Code (Sonnet 4.5)**, **Codex (GPT-5.2, GPT-5.1 mini)**, **Qwen Code (Qwen3-30b-coder)** 총 4종입니다. 동일한 태스크를 컨텍스트 파일이 있는 경우와 없는 경우로 나누어 성공률, 비용, 행동 패턴을 비교했습니다.

## 성공률 하락

논문에서 가장 주목할 결과는 **LLM이 자동 생성한 컨텍스트 파일**을 제공했을 때입니다.

```text
컨텍스트 파일 없음:            기준 성공률 (baseline)
LLM 생성 컨텍스트 파일 제공:   SWE-bench Lite에서 평균 0.5% 하락
                               AGENTbench에서 평균 2% 하락
```

수치 자체는 작아 보일 수 있지만, **도움이 될 것으로 기대한 파일이 오히려 성능을 깎아 먹는다**는 점에서 의미가 큽니다. 더 나은 결과를 위해 추가한 설정이 역효과를 내는 상황입니다.

## 불필요한 탐색과 도구 사용 증가

컨텍스트 파일이 제공되면 AI는 해당 정보를 성실하게 참고하려 합니다. 논문은 이 과정에서 발생하는 구체적인 행동 변화를 측정했습니다.

- 평균 실행 **스텝 수 증가**: SWE-bench에서 +2.45 스텝, AGENTbench에서 +3.92 스텝
- **추론 토큰 증가**: GPT-5.2 기준 SWE-bench에서 +22%, AGENTbench에서 +14%
- 컨텍스트 파일에 **특정 도구가 언급되면 해당 도구 사용량이 급증**: 예를 들어, `uv`가 언급된 경우 사용 빈도가 0.01x에서 1.6x로 증가

예를 들어, "이 함수의 버그를 수정해줘"라는 단순한 요청에도 컨텍스트 파일에 폴더 구조 설명이 있으면 AI가 해당 구조를 확인하려 이것저것 파일을 읽기 시작합니다. 정작 버그 수정에는 필요 없는 작업을 하면서 시간과 토큰을 낭비하는 것입니다.

## 비용 20% 이상 증가

```text
LLM 생성 컨텍스트 파일:
  SWE-bench Lite: 추론 비용 20% 증가
  AGENTbench:     추론 비용 23% 증가

개발자 작성 컨텍스트 파일:
  최대 19% 비용 증가
```

컨텍스트 파일의 내용 자체가 토큰을 차지하는 것은 물론, AI가 그 내용을 참고하여 더 많은 파일을 탐색하고 도구를 호출하면서 총 비용이 올라갑니다. 비용은 올라가는데 성공률은 떨어지니, 투자 대비 수익(ROI)이 마이너스인 셈입니다.

# 3. 사람이 작성한 컨텍스트 파일은 다르다

---

## 사람 vs LLM 생성 파일 비교

중요한 반전이 있습니다. **사람이 직접 작성한 컨텍스트 파일**은 LLM 자동 생성 파일과 다른 결과를 보입니다.

| 컨텍스트 파일 유형 | 성공률 변화 | 비용 변화 |
|-------------------|-----------|----------|
| LLM 자동 생성 | 0.5~2% **하락** | 20~23% **증가** |
| 개발자 직접 작성 | AGENTbench에서 평균 4% **상승** | 최대 19% 증가 |
| 없음 (baseline) | - | - |

단, 사람이 작성하더라도 **작업과 관련 없는 내용이 포함되면** 컨텍스트 파일이 없는 경우보다 성공률이 낮아질 수 있습니다. 논문의 결론에서도 이를 명확히 합니다: "Human-written context files should describe only minimal requirements."

## 왜 사람이 작성한 파일이 더 효과적인가?

LLM이 생성한 파일은 포괄적이지만 **필요 이상으로 광범위**합니다. 리포지토리의 모든 정보를 요약하려 하기 때문에, 특정 태스크에는 불필요한 내용이 대량으로 포함됩니다.

반면 사람이 작성한 파일은 실제로 중요하다고 판단한 정보만을 담습니다. 이 차이가 AI의 작업 집중도에 직접적인 영향을 줍니다.

이는 개발자가 AI를 더 잘 다루는 방법을 알고 있다는 반증이기도 합니다. **어떤 정보를 줘야 하는지 아는 것, 더 중요하게는 어떤 정보를 주지 말아야 하는지 아는 것**이 AI 활용 능력의 핵심입니다.

## 작업 무관 정보의 해악

논문에서는 "불필요한 요구사항이 태스크를 더 어렵게 만든다(unnecessary requirements make tasks harder)"고 결론짓습니다. 구체적인 예시를 들어보겠습니다.

```text
현재 작업: "API 응답에 페이지네이션 추가"

컨텍스트 파일에 있으면 해로운 정보 예시:
  - 프로젝트 전체 폴더 구조 설명  → AI가 관련 없는 디렉터리까지 탐색
  - 테스트 코드 작성 규칙         → 요청하지 않은 테스트까지 작성 시도
  - 배포 프로세스 설명             → 배포 관련 파일까지 확인
  - 과거에 사용하던 라이브러리 정보 → deprecated 패턴으로 코드 작성
```

이런 정보들이 AI의 목표 달성을 방해합니다. AI가 "이 규칙도 반영해야 하나?"라고 판단하여 불필요한 행동을 취하게 됩니다.

# 4. 반대 결과의 논문: 효율성이 올라간다?

---

같은 시기에 발표된 또 다른 논문도 함께 살펴볼 필요가 있습니다. **"On the Impact of AGENTS.md Files on the Efficiency of AI Coding Agents"** (Lulla et al., arXiv:2601.20404, 2026년 1월)는 오히려 긍정적인 결과를 보고합니다.

| 측정 항목 | 결과 |
|----------|------|
| 실행 시간(runtime) | 중앙값 **28.64% 감소** (98.57초 -> 70.34초) |
| 출력 토큰 소비 | 중앙값 **16.58% 감소** (2,925 -> 2,440 토큰) |
| 태스크 완료율 | 비슷한 수준 유지 |

10개 리포지토리, 124개 PR을 대상으로 OpenAI Codex 에이전트를 사용한 실험입니다. AGENTS.md가 있을 때 에이전트가 **더 빠르고 적은 토큰으로** 작업을 완료했다는 것입니다.

## 왜 결과가 다른가?

두 논문의 결과가 상반되는 이유를 분석하면 핵심 시사점이 드러납니다.

| 비교 항목 | Gloaguen et al. (부정적 결과) | Lulla et al. (긍정적 결과) |
|----------|-------------------------------|---------------------------|
| 컨텍스트 파일 | LLM 자동 생성 + 개발자 작성 | 개발자 작성 (실제 리포에 존재) |
| 벤치마크 | SWE-bench Lite, AGENTbench | 실제 PR 기반 |
| 태스크 규모 | 다양한 크기 | 소규모 (100줄 이하, 5개 파일 이하) |
| 측정 초점 | 성공률 + 비용 | 효율성 (시간, 토큰) |

핵심 차이는 **컨텍스트 파일의 품질**입니다. Lulla et al.의 실험에서 사용된 파일은 해당 리포지토리 개발자가 직접 작성한 것이었고, Gloaguen et al.의 SWE-bench 실험에서 부정적 결과를 보인 파일은 LLM이 자동 생성한 것이었습니다. Gloaguen et al.에서도 개발자 작성 파일(AGENTbench)은 평균 4% 성공률 상승을 보였다는 점을 기억해야 합니다.

**결론: 컨텍스트 파일 자체가 나쁜 것이 아니라, 잘못 만든 컨텍스트 파일이 나쁜 것입니다.**

# 5. "AI가 멍청해졌다"는 불만의 진짜 원인

---

최근 개발자들 사이에서 "AI가 예전보다 멍청해졌다", "응답이 느려졌다"는 인식이 퍼지고 있습니다. 두 논문의 결과를 종합하면 그 원인 중 하나로 **컨텍스트 파일의 비대화**를 지목할 수 있습니다.

프로젝트가 진행되면서 CLAUDE.md에 내용이 쌓이는 것은 자연스러운 일입니다. 새로운 컨벤션이 생기면 추가하고, 주의사항이 생기면 추가하다 보면 파일이 점점 커집니다.

```text
CLAUDE.md 초기:     핵심 내용 10줄   → AI가 집중해서 읽음 → 빠른 응답
CLAUDE.md 6개월 후: 잡다한 내용 300줄 → AI가 산만하게 행동 → 느린 응답, 비용 증가
```

논문에서 관측된 메커니즘을 이 상황에 대입하면 이렇습니다.

1. CLAUDE.md에 내용이 많아질수록 **추론 토큰이 10~22% 증가**한다
2. 불필요한 규칙이 많을수록 **AI가 추가적인 파일 탐색과 도구 호출**을 수행한다
3. 그 결과 응답 시간이 늘어나고, 핵심 태스크에 대한 집중도가 떨어진다

주기적으로 CLAUDE.md를 정리하지 않으면 오히려 AI 성능을 떨어뜨리는 역설이 발생합니다.

# 6. 실무 적용: 효율적인 컨텍스트 파일 관리 방법

---

두 논문의 결과를 종합하여 실무적인 관리 방법을 정리합니다.

## 6-1. 최소 필수 원칙

컨텍스트 파일을 작성할 때 다음 질문을 스스로 던져봅니다.

```text
"이 정보가 없으면 AI가 실수를 할 것인가?"

YES → 포함
NO  → 제외
```

단순히 "알면 좋겠다"는 수준의 정보는 넣지 않는 것이 낫습니다. 논문의 결론을 빌리면, **"minimal requirements만 기술하라"**입니다.

## 6-2. 정기적인 정리

CLAUDE.md는 **살아있는 문서**여야 합니다. 추가만 하는 것이 아니라 주기적으로 불필요한 내용을 제거해야 합니다.

```text
정리 체크리스트:
  [ ] 현재 프로젝트에서 더 이상 사용하지 않는 패턴이 있는가?
  [ ] 일반적인 프로그래밍 지식이 포함되어 있는가? (AI가 이미 알고 있는 것)
  [ ] 현재 작업 흐름과 무관한 히스토리 정보가 있는가?
  [ ] 한 번만 쓰고 다시 쓸 일 없는 특수 상황 설명이 있는가?
  [ ] "AI가 이 줄을 안 읽어도 대부분의 작업을 잘 수행하는가?"
```

해당하는 항목이 있다면 삭제 후보입니다.

## 6-3. 스킬(Skill)로 분리

Claude Code의 **스킬(Skill) 기능**을 활용하면 CLAUDE.md를 가볍게 유지하면서도 필요한 정보를 상황에 맞게 제공할 수 있습니다. 스킬은 `SKILL.md` 파일에 정의된 재사용 가능한 지시사항으로, `/skill-name` 형태의 슬래시 커맨드로 호출하거나 Claude가 태스크 컨텍스트에 따라 자동으로 호출합니다.

```text
CLAUDE.md에 남길 것 (항상 적용되는 핵심):
  - 빌드/실행 명령어
  - 절대 위반하면 안 되는 핵심 컨벤션
  - 팀 전원에게 항상 적용되는 최소한의 규칙

스킬로 분리할 것 (특정 작업 시에만 로드):
  - PR 리뷰 시 체크리스트
  - 블로그 포스트 작성 규칙
  - 특정 모듈 개발 시 참고할 아키텍처 설명
  - 테스트 코드 작성 가이드
```

이렇게 하면 AI는 현재 작업에 필요한 스킬이 호출될 때만 해당 정보를 받아서 처리합니다. 관련 없는 정보로 인한 노이즈가 줄어듭니다.

구체적인 디렉터리 구조와 예시는 다음과 같습니다.

```text
프로젝트 루트/
├── CLAUDE.md                            ← 항상 로드 (최소한으로 유지)
└── .claude/
    └── skills/
        ├── write-test/
        │   └── SKILL.md                 ← /write-test 호출 시에만 로드
        ├── review-pr/
        │   └── SKILL.md                 ← /review-pr 호출 시에만 로드
        └── write-blog/
            └── SKILL.md                 ← /write-blog 호출 시에만 로드
```

```markdown
# CLAUDE.md (최소화 - 항상 적용)
- 빌드: uv run python main.py
- 테스트: uv run pytest -v
- 커밋 시 반드시 /commit 스킬 사용
```

```markdown
# .claude/skills/write-test/SKILL.md (테스트 작성 시에만 로드)
- 테스트 파일명: test_*.py
- 한글 메서드명 사용 (예: test_설정값이_없으면_예외_발생)
- mock 주입 방식: 파라미터로 주입
- 테스트 프레임워크: pytest
```

이 방식의 핵심은 **논문에서 지적한 "작업 무관 정보" 문제를 구조적으로 해결**한다는 것입니다. 테스트 작성 규칙은 버그 수정 작업 중에는 로드되지 않고, PR 리뷰 체크리스트는 코드 작성 중에는 로드되지 않습니다.

## 6-4. LLM 생성 컨텍스트 파일을 그대로 쓰지 않는다

`claude /init`이나 AI에게 "CLAUDE.md를 작성해줘"라고 요청하면 편리하지만, 논문 결과가 보여주듯 **LLM이 생성한 파일은 포괄적이지만 과도하게 광범위**합니다. LLM 생성 결과물을 초안으로 활용하되, 반드시 사람이 리뷰하여 현재 작업에 불필요한 내용을 제거해야 합니다.

```text
LLM 생성 컨텍스트 파일 활용 프로세스:
  1. AI에게 초안 생성 요청
  2. 사람이 리뷰: "이 정보 없으면 AI가 실수하나?"
  3. NO인 항목 삭제
  4. 남은 내용도 간결하게 재작성
```

# 7. 정리

---

두 논문의 결과를 하나의 표로 종합합니다.

| 상황 | 성공률 | 비용 | 출처 |
|------|--------|------|------|
| LLM 자동 생성 컨텍스트 파일 | 0.5~2% **하락** | 20~23% **증가** | Gloaguen et al. |
| 개발자가 잘 작성한 컨텍스트 파일 | 평균 4% **상승** | 최대 19% 증가 | Gloaguen et al. |
| 개발자 작성 파일 (소규모 태스크) | 유지 | 실행 시간 28% **감소**, 토큰 16% **감소** | Lulla et al. |
| 작업 무관 정보가 포함된 파일 | 파일 없는 것보다 **나쁨** | 증가 | 양쪽 공통 |
| 주기적으로 정리된 최소한의 파일 | **최적** | **최적** | 양쪽 공통 권장 |

```text
컨텍스트 파일 관리 원칙:
  1. LLM이 생성한 컨텍스트 파일을 그대로 신뢰하지 않는다
  2. 사람이 직접 작성하되, 최소한의 필수 정보만 담는다 (minimal requirements)
  3. 주기적으로 정리하여 불필요한 내용을 제거한다
  4. 특정 작업에만 필요한 상세 정보는 스킬로 분리한다
```

CLAUDE.md는 길수록 좋은 것이 아닙니다. AI에게 모든 것을 알려주려 하기보다, **지금 작업에 꼭 필요한 것만 전달하는 것**이 논문이 검증한 최적의 방법입니다.
