# Introduction

---

Claude Code를 사용하다 보면 "You've reached your usage limit"라는 메시지를 마주치게 됩니다. 처음에는 당황스럽지만, 이 제한이 어떻게 작동하는지 이해하면 훨씬 효율적으로 사용할 수 있습니다.

이 글에서는 Claude Code의 사용량 제한 방식, 플랜별 차이, 제한에 걸렸을 때의 대응 방법, 그리고 토큰을 아끼는 실전 팁을 정리합니다. 공식 문서([code.claude.com](https://code.claude.com))를 기반으로 하되, 커뮤니티에서 보고된 비공식 정보는 별도로 표시하여 구분합니다.

# 1. 사용량 제한 방식

---

## 1-1. 롤링 윈도우 (Sliding Window) 방식

Claude Code의 사용량 제한은 **고정된 시간에 리셋되는 방식이 아니라 롤링 윈도우(sliding window) 방식**입니다.

```text
예시:

10:00에 큰 작업 수행 → 토큰 대량 사용
15:00에 제한 도달

이 경우 15:00부터 정확히 5시간 후에 전량 리셋되는 것이 아니라,
10:00의 사용량이 윈도우에서 빠지면서 점진적으로 사용 가능한 토큰이 늘어남
```

"매일 자정에 리셋"같은 고정 리셋 방식과 다르게, 사용한 시점 기준으로 점진적으로 사용 가능 토큰이 회복됩니다. 이 때문에 언제 정확히 사용 가능해질지 예측하기 어렵습니다.

## 1-2. 듀얼 레이어 제한

Claude Code는 **두 가지 레벨의 사용량 제한**을 적용합니다:

| 제한 유형 | 설명 |
|-----------|------|
| **단기 제한** | 급격한 사용량 폭증을 방지하는 짧은 주기의 롤링 윈도우 |
| **장기 제한** | 장기적인 과다 사용을 방지하는 더 긴 주기의 상한 |

단기 제한을 통과해도 장기 상한에 걸릴 수 있으므로, 지속적으로 많이 사용하면 결국 제한에 도달합니다.

> **비공식 정보**: "5시간 단기 윈도우", "7일 주간 상한"이라는 구체적 수치는 커뮤니티에서 보고된 비공식 정보입니다. 공식 문서([code.claude.com](https://code.claude.com), [support.anthropic.com](https://support.anthropic.com))에는 구체적인 윈도우 시간이 명시되어 있지 않으며, Anthropic이 언제든 변경할 수 있습니다.

## 1-3. 사용량 공유 범위

Claude Code의 사용량은 **같은 계정의 Claude 웹/앱 채팅과 합산**됩니다.

```text
동일 계정에서의 사용량 합산:

┌──────────────────────┐    ┌──────────────────────┐
│  Claude 웹/앱 채팅    │    │    Claude Code       │
│  (claude.ai)         │    │    (터미널)           │
└──────────┬───────────┘    └──────────┬───────────┘
           │                           │
           └──────────┬────────────────┘
                      ▼
              ┌──────────────┐
              │  합산 사용량    │
              │  (Rate Limit) │
              └──────────────┘
```

따라서 웹 채팅에서 긴 대화를 하면서 동시에 Claude Code로 코딩 작업을 하면, 예상보다 빠르게 제한에 도달할 수 있습니다.

# 2. 플랜별 사용량 차이

---

Claude는 3가지 유료 플랜을 제공하며, 각 플랜마다 사용량 한도가 다릅니다.

| 플랜 | 월 요금 | 상대적 사용량 | 특징 |
|------|---------|:------------:|------|
| **Pro** | $20 | 기본 (1x) | 일반적인 개인 사용자 대상 |
| **Max 5x** | $100 | ~5배 | 헤비 유저 대상 |
| **Max 20x** | $200 | ~20배 | 전문 개발자/팀 대상 |

공식 문서에서는 **"~5x", "~20x"**로 표현하며 정확한 토큰 수치를 공개하지 않습니다. "정확히 5배"가 아니라 "대략 5배 수준"으로 이해해야 합니다.

> **참고**: Free 플랜에서도 Claude Code를 사용할 수 있지만, 사용량이 매우 제한적입니다.

# 3. Fast Mode와 Rate Limit

---

Fast Mode는 Claude Code의 응답 속도를 높이는 옵션으로, **별도의 rate limit**을 가지고 있습니다.

## 3-1. Fast Mode란?

Fast Mode는 **별도의 모델이 아닙니다**. 동일한 Claude Opus 4.6 모델을 **더 빠른 출력 속도**로 사용하는 옵션입니다.

```bash
# Fast mode 토글 (활성화/비활성화)
/fast
```

활성화되면 프롬프트 영역에 `↯` 아이콘이 표시됩니다.

## 3-2. Fast Mode 제한 동작 방식

```text
1. Fast mode로 작업 진행 (↯ 아이콘 활성)
2. Fast mode rate limit 도달
3. 자동으로 표준 Opus 4.6 속도로 전환 (↯ 아이콘이 회색으로 변경)
4. 쿨다운 기간 경과 후 자동으로 fast mode 재활성화
```

핵심은 **Fast mode 제한에 걸려도 작업이 중단되지 않는다**는 것입니다. 표준 속도로 자동 전환되어 계속 진행됩니다. 쿨다운을 기다리지 않고 수동으로 fast mode를 비활성화하려면 `/fast` 명령을 다시 실행하면 됩니다.

## 3-3. Fast Mode 비용과 요구 사항

Fast Mode에는 중요한 비용 관련 사항이 있습니다.

| 항목 | 내용 |
|------|------|
| **비용** | 표준 Opus 4.6보다 **높은 per-token 가격** |
| **과금 방식** | 플랜 포함 사용량과 **별도로 extra usage로 과금** |
| **사용량 계산** | Fast mode 토큰은 **플랜 포함 사용량에 카운트되지 않음** |
| **필수 조건** | Extra usage가 활성화되어 있어야 함 |

> **주의**: Fast mode 토큰은 첫 토큰부터 별도의 fast mode 요율로 과금됩니다. 플랜에 잔여 사용량이 있어도 fast mode 사용분은 extra usage로 청구됩니다.

**호환되지 않는 환경:**
- Amazon Bedrock
- Google Vertex AI
- Microsoft Azure Foundry

Fast mode는 Anthropic Console API 또는 Claude 구독 플랜(Extra usage 활성화 필요)에서만 사용 가능합니다.

## 3-4. Mid-conversation 전환 시 비용 주의

대화 중간에 fast mode로 전환하면, **전체 대화 컨텍스트에 대해 fast mode 비처리 입력 토큰(uncached input token) 가격**이 적용됩니다. 이는 처음부터 fast mode를 활성화한 경우보다 비용이 더 높습니다.

```text
비용 비교:

처음부터 Fast mode ON:
  → 캐시된 입력 토큰 + fast mode 출력 토큰

대화 중간에 Fast mode ON:
  → 전체 컨텍스트를 uncached fast mode 가격으로 재처리
  → 비용이 더 높음
```

따라서 fast mode를 사용할 계획이라면 **세션 시작 시점에 활성화**하는 것이 비용 효율적입니다.

# 4. 사용량 확인 방법

---

Claude Code는 사용량을 확인할 수 있는 명령어를 제공합니다. 다만 구독 사용자(Pro/Max)와 API 사용자가 확인해야 할 명령어가 다릅니다.

## 4-1. /stats (구독 사용자용 - 사용 패턴 분석)

```bash
/stats
```

세션의 **사용 패턴**을 분석합니다. 도구 호출 빈도, 평균 응답 시간 등을 통해 어떤 작업이 토큰을 많이 소비하는지 파악할 수 있습니다.

**Pro/Max 구독 사용자**는 사용량이 구독에 포함되어 있으므로, `/stats`로 사용 패턴을 확인하는 것이 적절합니다.

## 4-2. /cost (API 사용자용 - 토큰/비용 확인)

```bash
/cost
```

현재 세션의 **API 토큰 사용량과 예상 비용**을 확인합니다.

> **공식 문서 기준**: `/cost` 명령은 API 토큰 사용량을 표시하며 **API 사용자를 위한 명령**입니다. Claude Max/Pro 구독 사용자는 사용량이 구독에 포함되어 있으므로, `/cost` 데이터가 과금 목적으로는 의미가 없습니다. 구독 사용자는 `/stats`를 사용하세요.

## 4-3. 사용량 확인 요약

| 명령어 | 대상 사용자 | 확인 내용 |
|--------|------------|-----------|
| `/stats` | Pro/Max 구독 사용자 | 사용 패턴, 도구 호출 빈도, 응답 시간 |
| `/cost` | API(pay-as-you-go) 사용자 | 토큰 사용량, 예상 비용 |

> **참고**: 두 명령어 모두 **정확한 할당량 잔여량**은 표시하지 않습니다. 대략적인 사용 추세를 파악하는 데 활용하세요.

# 5. 제한에 걸렸을 때 선택지

---

사용량 제한에 도달하면 다음 중 하나를 선택할 수 있습니다.

## 5-1. 기다리기

롤링 윈도우가 지나면서 점진적으로 사용 가능한 토큰이 회복됩니다. 긴급하지 않은 작업이라면 가장 경제적인 방법입니다.

## 5-2. 플랜 업그레이드

```text
Pro ($20) → Max 5x ($100) → Max 20x ($200)
```

장기적으로 많이 사용한다면 플랜 업그레이드를 고려합니다. 플랜 변경은 즉시 적용되며, 남은 기간에 대한 비용은 일할 계산됩니다.

## 5-3. Extra Usage 활성화

플랜을 유지하면서 **초과분을 유료로 사용**하는 옵션입니다. 한 달만 많이 쓰고 다시 정상 사용으로 돌아갈 경우 유용합니다.

- **개인 계정**: Console 빌링 설정에서 활성화
- **Teams/Enterprise**: 관리자가 조직 전체에 대해 활성화해야 함

> **참고**: Extra usage를 활성화하면 플랜 포함 사용량을 초과하는 부분이 자동으로 과금됩니다. 예상치 못한 비용이 발생할 수 있으므로 Console에서 사용량을 주기적으로 확인하세요.

## 5-4. API 크레딧 전환

Anthropic Console에서 pay-as-you-go 방식으로 전환하여 사용량만큼 과금되는 방식입니다. 정확한 사용량 제어가 필요하거나, 구독 제한 없이 사용하고 싶은 경우에 적합합니다.

# 6. 토큰 절약 실전 팁

---

토큰을 효율적으로 사용하면 같은 플랜 내에서 더 많은 작업을 수행할 수 있습니다. 공식 문서의 권장 사항과 실전 노하우를 함께 정리합니다.

## 6-1. /clear로 컨텍스트 초기화

```bash
/clear
```

태스크를 전환할 때 이전 대화 컨텍스트를 초기화합니다. 불필요한 히스토리가 누적되면 매 요청마다 이전 대화 전체가 입력 토큰으로 전송되므로, 새로운 작업을 시작할 때는 `/clear`로 깨끗한 상태에서 시작하는 것이 좋습니다.

```text
대화가 길어질수록 매 요청의 입력 토큰이 증가:

요청 1: [시스템 프롬프트] + [메시지 1]                    → 입력 토큰 적음
요청 5: [시스템 프롬프트] + [메시지 1~4 + 응답 1~4] + [메시지 5]  → 입력 토큰 많음

/clear 후:
요청 1: [시스템 프롬프트] + [새 메시지]                    → 입력 토큰 다시 적어짐
```

## 6-2. /compact로 선택적 요약

```bash
/compact
```

현재 컨텍스트를 요약하여 토큰을 절약합니다. `/clear`와 달리 **대화 내용을 완전히 삭제하지 않고 압축**하므로, 이전 맥락을 유지하면서 토큰을 줄일 수 있습니다.

보존할 내용을 지정할 수도 있습니다:

```text
예시:
/compact 후 "데이터베이스 스키마 관련 내용은 유지하고 나머지는 요약해줘"
```

**`/clear` vs `/compact` 사용 기준:**

| 상황 | 권장 명령 | 이유 |
|------|----------|------|
| 완전히 다른 작업으로 전환 | `/clear` | 이전 컨텍스트가 불필요 |
| 같은 작업 내 컨텍스트가 커진 경우 | `/compact` | 핵심 맥락은 유지하면서 토큰 절약 |

## 6-3. 적절한 모델 선택

모든 작업에 Opus 4.6을 사용할 필요는 없습니다. 작업의 복잡도에 따라 적절한 모델을 선택하면 토큰을 절약할 수 있습니다.

| 모델 | 적합한 작업 |
|------|------------|
| **Opus 4.6** | 복잡한 아키텍처 설계, 난이도 높은 버그 수정, 대규모 리팩토링, 다단계 추론 |
| **Sonnet 4.5** | 간단한 CRUD 작성, 테스트 코드 작성, 문서 작성, 코드 리뷰 |
| **Haiku** | 단순 서브에이전트 작업, 빠른 분류/판단 |

```bash
# 모델 전환 명령
/model sonnet     # Sonnet 4.5로 전환
/model opus       # Opus 4.6으로 전환
```

> **참고**: `/config`에서 기본 모델을 설정할 수도 있습니다. 공식 문서에서는 대부분의 코딩 작업에 Sonnet이 적합하며, Opus는 복잡한 작업에만 사용할 것을 권장합니다.

## 6-4. 미사용 MCP 서버 비활성화

MCP(Model Context Protocol) 서버는 시스템 프롬프트에 도구 정의를 추가하므로 토큰을 소비합니다. 사용하지 않는 MCP 서버가 활성화되어 있으면 매 요청마다 불필요한 토큰이 사용됩니다.

사용하지 않는 MCP 서버는 설정 파일에서 비활성화합니다:

```json
// .claude/settings.json 또는 프로젝트 설정
{
  "mcpServers": {
    "active-mcp": {
      "command": "npx",
      "args": ["-y", "active-mcp-server"]
    }
    // "unused-mcp": {...}  ← 주석 처리 또는 삭제하여 비활성화
  }
}
```

## 6-5. 구체적 프롬프트 작성

모호한 요청은 Claude가 여러 방향을 탐색하느라 불필요한 도구 호출과 토큰을 소비합니다.

**비효율적인 프롬프트:**
```text
"이 코드 개선해줘"
```

**효율적인 프롬프트:**
```text
"UserService.java의 findByEmail() 메서드에 null 체크를 추가하고,
예외 처리를 CustomUserNotFoundException으로 변경해줘"
```

구체적인 프롬프트가 좋은 이유:
- Claude가 탐색해야 할 파일 범위가 줄어듦 (= 도구 호출 감소)
- 불필요한 방향 탐색 없이 바로 작업 수행 (= 추론 토큰 감소)
- 한 번에 원하는 결과를 얻을 확률이 높음 (= 재시도 감소)

## 6-6. 작업 범위 명확히 하기

여러 파일을 동시에 수정해야 할 때, 작업 범위를 명확히 하면 불필요한 컨텍스트 로딩을 줄일 수 있습니다.

**비효율적인 프롬프트:**
```text
"전체 프로젝트를 분석해서 개선점을 찾아줘"
```

**효율적인 프롬프트:**
```text
"UserService.java의 성능 문제만 찾아줘.
다른 서비스 클래스는 분석하지 않아도 돼"
```

넓은 범위의 요청은 Claude가 프로젝트 전체를 탐색하면서 대량의 토큰을 소비합니다. 필요한 범위만 지정하면 동일한 결과를 훨씬 적은 토큰으로 얻을 수 있습니다.

# 7. 흔한 실수와 주의사항

---

## 7-1. "자정에 리셋되겠지" 기대

롤링 윈도우 방식이므로 특정 시각에 전량 리셋되지 않습니다. 자정까지 기다린다고 해서 사용량이 초기화되는 것이 아니라, 과거 사용량이 윈도우에서 빠지는 만큼만 점진적으로 회복됩니다.

## 7-2. 웹 채팅과 Claude Code를 동시에 과다 사용

동일 계정의 사용량이 합산되므로, 웹에서 긴 대화를 나누면서 Claude Code로 대규모 리팩토링을 동시에 진행하면 예상보다 빠르게 제한에 도달합니다. 제한이 가까울 때는 한쪽에 집중하는 것이 효율적입니다.

## 7-3. /cost로 구독 잔여량 확인 시도

`/cost`는 API 사용자를 위한 명령입니다. Pro/Max 구독 사용자는 `/cost`가 아닌 `/stats`로 사용 패턴을 확인해야 합니다. `/cost`로 표시되는 금액은 구독 사용자의 실제 과금과 무관합니다.

## 7-4. 대화 중간에 Fast Mode 활성화

대화가 길어진 상태에서 fast mode로 전환하면, 전체 대화 컨텍스트에 uncached fast mode 가격이 적용되어 비용이 크게 증가할 수 있습니다. Fast mode를 사용할 계획이라면 세션 시작 시 활성화하세요.

## 7-5. 컨텍스트 관리 없이 장시간 사용

대화가 길어질수록 매 요청의 입력 토큰이 누적 증가합니다. 작업이 전환될 때마다 `/clear` 또는 `/compact`로 컨텍스트를 관리하지 않으면, 동일한 작업에도 점점 더 많은 토큰이 소비됩니다.

# 8. 정리

---

| 항목 | 내용 |
|------|------|
| **제한 방식** | 롤링 윈도우 (단기 + 장기 듀얼 레이어) |
| **공유 범위** | Claude 웹/앱 채팅 + Claude Code 합산 |
| **플랜** | Pro ($20, 1x), Max 5x ($100, ~5x), Max 20x ($200, ~20x) |
| **제한 시 대응** | 기다리기 / 플랜 업그레이드 / Extra Usage / API 전환 |
| **사용량 확인** | 구독 사용자: `/stats` / API 사용자: `/cost` |
| **Fast mode** | 별도 rate limit, extra usage로 과금, 제한 시 표준 모드 자동 전환 |
| **절약 팁** | `/clear`, `/compact`, 모델 선택(`/model`), MCP 비활성화, 구체적 프롬프트, 작업 범위 제한 |

**핵심 원칙:**

- **롤링 윈도우 방식**이므로 특정 시각 리셋을 기대하지 말 것 (구체적 윈도우 시간은 비공식 정보)
- **구독 사용자는 `/stats`**, API 사용자는 `/cost`로 사용량 확인
- **구체적 프롬프트와 적절한 모델 선택**이 가장 효과적인 절약 전략
- **Fast mode는 extra usage로 별도 과금**되므로 비용 구조를 이해하고 사용할 것
