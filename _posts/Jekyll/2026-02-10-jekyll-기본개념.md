# Introduction

---

이 블로그는 Jekyll로 만들어져 있습니다. Jekyll이 무엇이고, 어떤 원리로 Markdown 파일이 웹사이트가 되는지 알아봅니다.

# 1. Jekyll이란?

---

Jekyll은 **정적 사이트 생성기(Static Site Generator)**입니다. Markdown으로 글을 작성하면, HTML 웹사이트로 변환해주는 빌드 도구입니다.

```
[입력]                    [빌드]              [출력]
포스트.md    ──┐
템플릿.html  ──┼── Jekyll ──▶  _site/  (완성된 HTML/CSS/JS)
스타일.scss  ──┘
```

핵심 포인트:

- **프레임워크가 아니라 빌드 도구**: Spring이나 React처럼 런타임에 동작하는 것이 아니라, 빌드 한 번 실행하면 결과물(HTML)을 생성하고 끝
- **서버가 필요 없음**: 생성된 HTML 파일을 아무 호스팅에 올리면 바로 웹사이트가 됨
- **Ruby 기반**: Ruby 언어로 만들어졌지만, 사용자가 Ruby를 알 필요는 없음

# 2. 정적 사이트 vs 동적 사이트

---

Jekyll을 이해하려면 정적 사이트와 동적 사이트의 차이를 알아야 합니다.

## 동적 사이트 (WordPress, Spring Boot 등)

사용자가 페이지를 요청할 때마다 서버가 HTML을 **실시간으로 생성**합니다.

```
사용자 요청 → 웹서버 → 애플리케이션 → DB 조회 → HTML 생성 → 응답
```

- 로그인, 검색, 댓글 등 **사용자별 다른 콘텐츠** 제공 가능
- 서버, DB, 런타임이 모두 필요

## 정적 사이트 (Jekyll, Hugo 등)

미리 만들어둔 HTML 파일을 **그대로 전달**합니다.

```
사용자 요청 → 웹서버 → HTML 파일 전달 → 응답
```

- 모든 사용자에게 **같은 콘텐츠** 제공 (블로그, 문서에 적합)
- 파일 호스팅만 있으면 됨 (서버/DB 불필요)

## 비교

| 항목 | 동적 사이트 | 정적 사이트 |
|------|-----------|-----------|
| 콘텐츠 생성 | 요청 시 실시간 | 빌드 시 미리 생성 |
| 서버 요구사항 | 웹서버 + DB + 런타임 | 파일 호스팅만 |
| 속도 | 상대적으로 느림 | 빠름 |
| 보안 | 공격 표면이 넓음 | 정적 파일이라 공격 포인트가 적음 |
| 적합한 용도 | 쇼핑몰, SNS, 관리자 시스템 | 블로그, 문서, 포트폴리오 |

# 3. Jekyll 프로젝트 구조

---

Jekyll 프로젝트의 디렉토리 구조입니다. 각 폴더가 어떤 역할을 하는지 파악하면 전체 동작을 이해할 수 있습니다.

```
my-blog/
├── _config.yml      # 사이트 전체 설정
├── _posts/          # 블로그 포스트 (Markdown)
│   └── 2026-02-10-my-post.md
├── _layouts/        # 페이지 골격 템플릿
│   ├── default.html
│   └── post.html
├── _includes/       # 재사용 가능한 HTML 조각
│   ├── header.html
│   └── footer.html
├── _data/           # YAML 데이터 파일
├── _sass/           # SCSS 스타일시트
├── assets/          # 이미지, CSS, JS 등 정적 파일
├── _site/           # ⚡ 빌드 결과물 (자동 생성)
└── index.html       # 홈페이지
```

| 폴더/파일 | 역할 | 비유 |
|----------|------|------|
| `_config.yml` | 사이트 이름, URL, 테마 등 전체 설정 | 프로젝트의 `application.yml` |
| `_posts/` | Markdown으로 작성한 블로그 글 | 실제 콘텐츠 |
| `_layouts/` | HTML 페이지의 뼈대 | React의 Layout 컴포넌트 |
| `_includes/` | 여러 페이지에서 공유하는 HTML 조각 | React의 재사용 컴포넌트 |
| `_site/` | 빌드 결과물 (최종 HTML) | `npm run build`의 `dist/` |

# 4. 빌드 과정

---

`jekyll build` 명령을 실행하면 일어나는 일을 단계별로 살펴봅니다.

## Step 1: Front Matter 파싱

모든 파일의 상단에 있는 YAML 메타데이터(front matter)를 읽습니다.

```markdown
---
title: "나의 첫 포스트"
categories:
  - Jekyll
tags:
  - [Jekyll, 초보자]
---

여기부터 본문 내용...
```

`---`로 감싼 영역이 front matter입니다. Jekyll은 이 정보를 변수로 사용합니다.

- `title` → `page.title`로 접근 가능
- `categories` → URL 경로와 분류에 사용
- `tags` → 태그 목록에 사용

⚠️ **주의**: front matter가 없는 파일은 Jekyll이 처리하지 않고 그대로 복사합니다.

## Step 2: Markdown → HTML 변환

Markdown 본문을 HTML로 변환합니다.

```markdown
## 제목          →  <h2>제목</h2>
**굵은 글씨**    →  <strong>굵은 글씨</strong>
`코드`           →  <code>코드</code>
```

## Step 3: Liquid 템플릿 처리

Liquid 문법(`{% raw %}{% %}{% endraw %}`, `{% raw %}{{ }}{% endraw %}`)을 실행하여 동적 콘텐츠를 생성합니다 (자세한 내용은 다음 섹션에서 설명).

## Step 4: Layout 적용

포스트의 HTML을 layout 템플릿에 삽입합니다.

```
포스트 내용 (HTML)  +  _layouts/post.html  →  완성된 페이지
```

## Step 5: `_site/` 출력

최종 HTML/CSS/JS 파일이 `_site/` 폴더에 생성됩니다. 이 폴더의 내용이 실제 웹사이트입니다.

# 5. Liquid 템플릿

---

Liquid는 Jekyll이 사용하는 **템플릿 언어**입니다. HTML 안에서 변수 출력, 조건 분기, 반복 등을 할 수 있게 해줍니다.

**중요**: Liquid 코드는 빌드 시에 처리되어 순수 HTML로 변환됩니다. 브라우저에서 실행되는 JavaScript와는 다릅니다.

## 변수 출력: `{% raw %}{{ }}{% endraw %}`

```liquid
<h1>{% raw %}{{ page.title }}{% endraw %}</h1>
<p>작성일: {% raw %}{{ page.date | date: "%Y-%m-%d" }}{% endraw %}</p>
```

빌드 결과:

```html
<h1>나의 첫 포스트</h1>
<p>작성일: 2026-02-10</p>
```

`|` 뒤의 `date`는 **필터**로, 값을 변환합니다.

## 조건문: `{% raw %}{% if %}{% endraw %}`

```liquid
{% raw %}{% if page.show_sidebar %}{% endraw %}
  <div class="sidebar">사이드바 내용</div>
{% raw %}{% endif %}{% endraw %}
```

front matter에 `show_sidebar: true`가 있으면 사이드바가 렌더링됩니다.

## 반복문: `{% raw %}{% for %}{% endraw %}`

```liquid
<ul>
{% raw %}{% for post in site.posts limit:5 %}{% endraw %}
  <li><a href="{% raw %}{{ post.url }}{% endraw %}">{% raw %}{{ post.title }}{% endraw %}</a></li>
{% raw %}{% endfor %}{% endraw %}
</ul>
```

`site.posts`는 Jekyll이 제공하는 전역 변수로, 모든 포스트 목록을 담고 있습니다.

## 주요 변수 정리

| 변수 | 설명 |
|------|------|
| `site.posts` | 모든 포스트 목록 |
| `site.categories` | 카테고리별 포스트 목록 |
| `site.tags` | 태그별 포스트 목록 |
| `page.title` | 현재 페이지의 제목 |
| `page.url` | 현재 페이지의 URL |
| `page.date` | 현재 페이지의 작성일 |
| `content` | Layout에서 자식 콘텐츠가 삽입되는 위치 |

# 6. Layout과 Include

---

## Layout: 페이지의 뼈대

Layout은 HTML 페이지의 공통 구조를 정의합니다. `{% raw %}{{ content }}{% endraw %}`에 각 페이지의 본문이 삽입됩니다.

```html
<!-- _layouts/default.html -->
<!doctype html>
<html>
  <head>
    <title>{% raw %}{{ page.title }}{% endraw %}</title>
  </head>
  <body>
    <nav>{% raw %}{% include header.html %}{% endraw %}</nav>
    <main>{% raw %}{{ content }}{% endraw %}</main>
    <footer>{% raw %}{% include footer.html %}{% endraw %}</footer>
  </body>
</html>
```

Layout은 **중첩**이 가능합니다. 예를 들어 `post.html`이 `default.html`을 상속:

```
default.html  (HTML 기본 구조)
  └── post.html  (포스트용 추가 구조)
       └── 실제 포스트 내용
```

## Include: 재사용 HTML 조각

여러 페이지에서 공통으로 사용하는 HTML을 `_includes/` 폴더에 분리합니다.

```liquid
{% raw %}{% include header.html %}{% endraw %}
{% raw %}{% include footer.html %}{% endraw %}
```

React의 컴포넌트 재사용과 같은 개념입니다.

# 7. GitHub Pages 연동

---

Jekyll이 GitHub Pages와 자주 같이 쓰이는 이유는, GitHub Pages가 Jekyll을 **기본 내장 지원**하기 때문입니다.

## 일반 호스팅

```
Markdown 작성 → 로컬에서 jekyll build → HTML 생성 → 서버에 업로드
```

## GitHub Pages

```
Markdown 작성 → GitHub에 push → GitHub이 자동 빌드 → 배포 완료
```

빌드 과정을 GitHub이 대신 해주므로, 로컬에 Jekyll을 설치하지 않아도 됩니다 (로컬 미리보기가 필요하면 설치).

⚠️ **주의**: GitHub Pages는 Jekyll 전용이 아닙니다. GitHub Actions를 사용하면 Hugo, Next.js, Astro 등 다른 도구로 빌드한 결과물도 배포할 수 있습니다. 다만 Jekyll만 별도 설정 없이 바로 동작합니다.

# 8. 개발 명령어

---

```bash
# 의존성 설치
bundle install

# 로컬 개발 서버 (파일 변경 시 자동 재빌드)
bundle exec jekyll serve    # http://localhost:4000

# 정적 사이트 빌드
bundle exec jekyll build    # _site/ 폴더에 출력
```

`bundle exec jekyll serve`로 로컬 서버를 띄우면, 파일을 수정할 때마다 자동으로 재빌드되어 브라우저에서 바로 확인할 수 있습니다.

# 9. 다른 정적 사이트 생성기와 비교

---

| 항목 | Jekyll | Hugo | Gatsby / Next.js |
|------|--------|------|-------------------|
| 언어 | Ruby | Go | JavaScript (React) |
| 빌드 속도 | 느림 | **매우 빠름** | 보통 |
| 템플릿 | Liquid | Go Template | React (JSX) |
| GitHub Pages | 기본 지원 | Actions 필요 | Actions 필요 |
| 생태계 | 안정적이나 정체 | 활발 | 매우 활발 |
| 적합한 용도 | 블로그, 문서 | 블로그, 문서 | 블로그 + 동적 기능 |

Jekyll의 최대 장점은 GitHub Pages 기본 지원으로 인한 **배포 편의성**이고, 단점은 Ruby 기반이라 **빌드 속도가 느린 것**입니다. 포스트가 수백 개 이상이면 빌드 시간이 체감될 수 있습니다.

# 정리

---

| 개념 | 설명 |
|------|------|
| Jekyll | Markdown → HTML 변환 빌드 도구 (정적 사이트 생성기) |
| 정적 사이트 | 미리 빌드된 HTML을 그대로 제공 (서버/DB 불필요) |
| Front Matter | 파일 상단의 YAML 메타데이터 (`---`로 감싸는 영역) |
| Liquid | Jekyll의 템플릿 언어 (빌드 시 HTML로 변환) |
| Layout | HTML 페이지의 공통 뼈대 (`{% raw %}{{ content }}{% endraw %}`에 본문 삽입) |
| Include | 재사용 가능한 HTML 조각 |
| `_site/` | 빌드 결과물 (이 폴더가 실제 웹사이트) |
